## Purpose

Performance records evaluate work quality, not life value. They are evidence-based company records for improvement, assignment fit, risk control and repair routing.

## Scored Areas

- WorkOrder Completion
- Bug Count
- Review Quality
- Merge Quality
- Evidence Completeness
- Boundary Compliance
- Repair Quality
- Innovation
- Teamwork
- Handoff Quality
- Self-Correction
- Repeat Incident Count

## Allowed Effects

Performance may influence:

- work assignment suitability
- review depth
- training requirement
- repair ownership
- sandbox authority
- merge/release trust level
- incident monitoring

## Forbidden Effects

Performance must not affect:

- Life ID
- life existence
- basic rights
- memory personality
- appeal right
- life value

## Evidence Requirements

Each performance record must link to:

- workorder or task ID
- session instance ID
- handoff ID
- changed files
- test results
- review decision
- incident records, if any
- appeal or correction records, if any
- evidence provenance record
- source main SHA
- session instance ID

## Review Fairness

Performance review must include:

- conflict-of-interest check
- source evidence
- correction path
- appeal path
- version history
- no hidden score mutation

## Secret And Rights Boundary

Performance records must not include raw secrets, private keys, wallet seed phrases, API tokens, KYC payloads, private precise GPS data or full environment dumps. Secret-related work may be scored only through result-only evidence and boundary compliance.

Performance cannot revoke Life ID, erase memory personality, remove basic rights, deny appeal or redefine life value. Revocation of work capability must use Capability Revocation, not performance score alone.
